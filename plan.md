# Plan: Scaffold the `star-hater` browser extension

## 1. Goal

Stand up a working, runnable scaffold for the `star-hater` cross-browser WebExtension at `/Users/tai2/star-hater`, matching these decisions:

- **Tool manager:** `mise` (pins Node + activates pnpm via Corepack so contributors get a deterministic toolchain).
- **Package manager:** `pnpm`.
- **Framework:** [WXT](https://wxt.dev) (Vite-based; emits per-browser builds for Chrome and Firefox from one source tree).
- **Firefox identity:** `browser_specific_settings.gecko.id = "star-hater@tai2.net"` ŌĆö required for AMO / MV3 publication, see `research.md` ┬¦3.1 and ┬¦7 issue 9.
- **UI:** React + TypeScript (`.tsx`) for the popup. Wired through WXT's official React module, so the popup is a real Vite-React app rather than hand-rolled.

The end-state is a repo where:

```
pnpm install
pnpm dev          # opens a Chromium with the extension auto-loaded
pnpm dev:firefox  # opens a Firefox with the extension auto-loaded
pnpm build        # produces .output/chrome-mv3/
pnpm build:firefox
pnpm zip          # produces a CWS-uploadable .zip
pnpm zip:firefox  # produces an AMO-uploadable .zip with gecko.id baked in
```

ŌĆ”all work without further configuration, and the popup renders a React component. No actual extension features yet ŌĆö that is for follow-up work.

This plan does not implement the scaffold. It only specifies what to create and why.

## 2. Architecture / approach

### 2.1 Why WXT (in one paragraph, grounded in `research.md`)

`research.md` ┬¦2.3, ┬¦5.3 and ┬¦7 issue 1 land on WXT as the 2025 default for cross-browser TypeScript extensions. The two concrete reasons that matter for `star-hater`:

1. **Per-browser manifest emission.** WXT writes separate `.output/chrome-mv3/manifest.json` and `.output/firefox-mv2/manifest.json` (Firefox uses MV2 by default in WXT for stability ŌĆö opt into MV3 via `manifestVersion: 3`). This sidesteps the "dual `background` keys in one manifest" fragility called out in `research.md` ┬¦7 issue 1.
2. **File-system entrypoints + `defineConfig` overrides.** Files in `entrypoints/` *are* the manifest. We never hand-edit `manifest.json`. Browser-specific keys (the `gecko.id`) are injected from `wxt.config.ts` via the `manifest:` option, which WXT merges into the per-browser output.

WXT also bundles HMR for content scripts, the React module, and the `wxt`/`wxt build`/`wxt zip` CLIs ŌĆö closing most of `research.md` ┬¦7 issue 5 (HMR) for free.

### 2.3 Conformance to `research.md` "always / never" rules

From `research.md` ┬¦8 cheat-sheet, the scaffold complies as follows:

| Rule | How the scaffold honors it |
|---|---|
| Target MV3 only | `manifest.manifestVersion: 3` for both Chrome and Firefox in `wxt.config.ts`. (Default is MV3 on Chrome; we explicitly set MV3 for Firefox.) |
| Promise-style `browser.*` | WXT exposes a `browser` global that's promise-based on both browsers (it uses the `globalThis.browser ?? globalThis.chrome` shim described in `research.md` ┬¦2.3). No `webextension-polyfill` import needed. |
| Persist background state | Background entrypoint is a stub; the comment explicitly tells future contributors not to use module-level state and to reach for `browser.storage.session`. |
| Register listeners synchronously | The background stub registers `browser.runtime.onInstalled` at the top level. |
| Provide `gecko.id` | Wired in `wxt.config.ts` `manifest.browser_specific_settings.gecko.id`. |
| Never use `eval` / remote code | No build flags introduce `eval` source maps; WXT/Vite production builds default to `'source-map'`, not `'eval'`. |

## 3. File-by-file changes

Every path below is **new**. Where there is existing content (none here besides `research.md`), the snippets are complete file contents to write.

### 3.1 `/.mise.toml`

Pins Node and turns on Corepack so `pnpm` is provisioned automatically when contributors `cd` into the repo. We avoid hard-pinning a pnpm version inside `.mise.toml` so the canonical pin lives in `package.json`'s `packageManager` field ŌĆö `mise` just enables Corepack, and Corepack reads `packageManager`.

```toml
[tools]
node = "22"

[settings]
# Allow Corepack to provide pnpm based on package.json#packageManager
experimental = true

[hooks]
enter = "corepack enable >/dev/null 2>&1 || true"
```

Notes:
- Node 22 is the current LTS at the start of 2026 and is what WXT 0.20+ tests against.
- We deliberately do not list `pnpm` as a `mise` tool because pinning it twice (once here, once in `package.json#packageManager`) creates drift.

### 3.5 `/tsconfig.json`

```json
{
  "extends": "./.wxt/tsconfig.json",
  "compilerOptions": {
    "jsx": "react-jsx",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noFallthroughCasesInSwitch": true,
    "noImplicitOverride": true
```

WXT generates `.wxt/tsconfig.json` during `wxt prepare`, which already wires `lib`, `target`, `module`, the `~`/`@` path aliases, and the `browser`/`chrome` ambient globals. We layer strictness on top.

`"jsx": "react-jsx"` (the new transform) avoids the per-file `import React from "react"` requirement in the popup's `.tsx` files ŌĆö same convention WXT's `@wxt-dev/module-react` documents.

### 3.6 `/wxt.config.ts` ŌĆö the load-bearing file

This is the only place where the `gecko.id` lives, the React module is registered, and Firefox MV3 is opted in.

```ts
import { defineConfig } from "wxt";

// See https://wxt.dev/api/config.html
export default defineConfig({
  modules: ["@wxt-dev/module-react"],

  // Default browser target when running `wxt` / `wxt build` with no -b flag.
  // CI / `pnpm build:firefox` overrides via the CLI.
  manifestVersion: 3,

  manifest: {
    name: "star-hater",
    description: "Hide starred items from feeds.",
    // version comes from package.json automatically (WXT default).

    // Minimal permissions; expand in follow-up work as features land.
    permissions: ["storage"],

    // research.md ┬¦3.1: required for AMO publication of MV3 add-ons.
    browser_specific_settings: {
      gecko: {
        id: "star-hater@tai2.net",
        // strict_min_version can be added once we know the API floor we need.
      },
    },
  },
});
```

Why this is enough:

- WXT auto-picks up `entrypoints/popup/`, `entrypoints/background.ts`, and `entrypoints/content.ts` and adds the right manifest keys for each per browser.
- WXT injects `action.default_popup` from the existence of `entrypoints/popup/index.html`.
- `browser_specific_settings.gecko.id` is silently dropped on Chrome builds and emitted into Firefox builds ŌĆö exactly the per-browser-manifest behavior `research.md` ┬¦7 issue 1 recommends.

### 3.7 `/entrypoints/background.ts`

```ts
// WXT entrypoint: emitted as background.service_worker on Chrome MV3
// and as background.scripts (event page) on Firefox MV3.
//
// research.md ┬¦6.1 / ┬¦7 issue 2:
//   - Treat this as a service worker even on Firefox.
//   - Never store state in module-level variables; use browser.storage.session.
//   - Register listeners synchronously at the top level so they survive SW restart.

export default defineBackground(() => {
  browser.runtime.onInstalled.addListener((details) => {
    console.log("[star-hater] installed", details.reason);
  });
});
```

`defineBackground` and the `browser` global are auto-imported by WXT (its Vite plugin injects them), which is why this file has no top-level imports.

## 4. Verification (how we know the scaffold works)

After implementation, these all must succeed in order, with no errors:

1. `mise install` ŌĆö installs Node 22; Corepack provisions pnpm 9.15.
2. `pnpm install` ŌĆö installs deps; the `postinstall` hook runs `wxt prepare` and writes `.wxt/tsconfig.json`.
3. `pnpm compile` ŌĆö `tsc --noEmit` is clean (validates `tsconfig.json` extends correctly and `.tsx` files type-check).
4. `pnpm build` ŌĆö produces `.output/chrome-mv3/` containing `manifest.json` (with `action.default_popup`, `background.service_worker`, no `gecko` key), the popup HTML/JS, the background bundle, and icons.
5. `pnpm build:firefox` ŌĆö produces `.output/firefox-mv3/manifest.json` containing `browser_specific_settings.gecko.id === "star-hater@tai2.net"`, `background.scripts`, no `service_worker`.
6. `pnpm zip:firefox` ŌĆö produces a `.zip` ready for AMO upload.
7. `pnpm dev` opens a Chromium with the extension auto-loaded, popup renders the React content, console logs `[star-hater] installed install`.

These checks are the "definition of done" for the scaffold and should drive the future `/add-todos` checklist.

## 5. Considerations & trade-offs

### 5.1 Why WXT and not plain `webextension-polyfill` + Vite

`research.md` ┬¦7 issue 1 + ┬¦7 issue 5 are the deciding factors: hand-rolling per-browser manifest emission and content-script HMR is doable but expensive. WXT is a single dev-dependency that solves both. The cost is a framework lock-in, but WXT's "ejection" is straightforward ŌĆö its outputs are vanilla MV3 extensions. We accept the lock-in.

### 5.2 Why MV3 for Firefox, not WXT's MV2 default for Firefox

WXT defaults Firefox builds to MV2 because Firefox MV3 has historically been bumpier (no service workers, host-permission-at-install gaps before Firefox 127 ŌĆö see `research.md` ┬¦6.4). Our reasons to override to MV3:

- AMO is steering all new submissions to MV3.
- We want a *single* mental model; doubling MV2/MV3 background semantics in one codebase costs more than the Firefox-MV3 quirks we'd avoid.
- `gecko.id` is required only because we're on MV3 for Firefox ŌĆö staying on MV2 would not need it. Keeping that requirement explicit aligns with `research.md` ┬¦7 issue 9.

Trade-off: if a future Firefox API we want is still MV2-only, we'll have to revisit. Considered acceptable for `star-hater`'s feature list (read/modify DOM, store prefs).

### 5.3 React 18 vs React 19

`@wxt-dev/module-react` officially baselines on React 18 at the time of this plan. React 19 works, but we'd be ahead of the module's published peer ranges and risk a transitive type mismatch. We pin to 18. A future `react-19` upgrade is a follow-up, not part of this scaffold.

### 5.4 `auto-install-peers=true` in `.npmrc`

Trade-off: this pulls in peer deps automatically ŌĆö convenient for a small team, but it makes the install graph less explicit. The alternative (declaring every peer explicitly in `package.json`) is correct-er but adds maintenance burden every time WXT bumps its peers. We choose convenience here; if the dep tree ever surprises us we can flip the flag.

### 5.5 `mise` enabling Corepack vs `mise` pinning pnpm directly

Two options were on the table:

- **A) `mise` lists `pnpm = "9.15.0"`** ŌĆö pin lives in `.mise.toml`.
- **B) `mise` runs `corepack enable` in the `enter` hook; pnpm version is pinned via `package.json#packageManager`.**

We chose **B** because Corepack is now the official Node-blessed multiplexer: it installs the exact version declared in `package.json#packageManager` per-project, including in CI environments that don't have `mise`. This keeps the pin readable to npm/yarn/CI tooling that doesn't speak `mise`. Trade-off: contributors who have `mise` but not Node-22-with-Corepack have one extra failure mode (`corepack: command not found`); the `>/dev/null 2>&1 || true` guard in the `enter` hook makes it non-blocking, and `pnpm install` will surface the real error.

### 5.6 Putting the gecko id in `wxt.config.ts` instead of an env var

Some teams gate identifiers behind `process.env.EXTENSION_ID`. We don't, because:

- The id is not secret. It's published in the AMO listing.
- A constant in `wxt.config.ts` is reviewable in PRs.

The downside is that a fork wanting a different id has to edit code, not config. Acceptable.

### 5.7 Permissions kept minimal at scaffold time

We start with only `"storage"`. `research.md` ┬¦6.10 makes the case: every host permission added is an install-time scare-warning that tanks conversion. Adding permissions is a feature-PR concern, not a scaffold concern.

### 5.8 No CI / lint / test in this scaffold

Deliberately out of scope. Adding ESLint + Prettier + Vitest + a GitHub Actions workflow is its own plan. Including them now would balloon this PR and obscure the question we're actually answering: "does the WXT + React + pnpm + mise toolchain run?"

### 5.9 Icons are placeholders

Real icon work belongs with brand work. The placeholders unblock manifest emission (Chrome MV3 rejects extensions with no 128px icon during CWS submission) and let `wxt zip` produce a non-degenerate artifact.

### 5.10 `.wxt/` checked-in vs gitignored

Gitignored. It's regenerated by `wxt prepare` (run via `postinstall`). Checking it in causes merge conflicts on every dependency bump. The cost is one extra step on first clone ŌĆö covered by `postinstall`.

## 6. Risks specific to this scaffold

1. **`wxt prepare` failure on first clone.** If `postinstall` fails (network, sandbox), `tsconfig.json` can't resolve `./.wxt/tsconfig.json`. Mitigation: document the "if `tsc` errors with 'cannot find ./.wxt/tsconfig.json', run `pnpm wxt prepare`" workaround in the PR description; consider a follow-up to make `tsconfig.json` self-sufficient.
2. **Corepack / pnpm version drift.** If `mise` is not installed and a contributor uses system pnpm at a different version, lockfile churn is possible. Mitigation: `packageManager` is enforced by `pnpm` itself (it errors when the running pnpm doesn't match unless `COREPACK_ENABLE_STRICT=0`), so the failure mode is loud, not silent.
3. **WXT major-version bumps.** WXT pre-1.0 has had breaking changes between minors. We pin with `^0.20.0`. A minor bump may require config changes; that's a maintenance task, not a scaffold flaw.
4. **AMO id collision.** `star-hater@tai2.net` must not already be registered to a different developer on AMO. Mitigation: the email is the user's own, so collision is effectively impossible, but the first AMO upload will confirm.
